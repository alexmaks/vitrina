# ТЗ: «Витрина-в-кармане» — этап 1 (PWA-админка + регистрация)

## Цель и контекст

Расширить существующий статический прототип (этап 0) до полноценного веб-приложения, в котором мастер сам регистрируется через Telegram, создаёт свою витрину, наполняет её товарами и получает публичную ссылку. Никакого нативного приложения — только PWA, чтобы мастер сохранил иконку на главный экран телефона.

## Что уже есть (этап 0)

В текущем коде:
- Next.js 14+ с App Router, TypeScript, Tailwind
- Статическая генерация витрин из YAML-файлов в `data/merchants/`
- Компоненты витрины: `StorefrontHeader`, `SaleBanner`, `ProductGrid`, `ProductCard`, `ContactButton`
- Yandex Metrika с условным рендером
- OG-картинки, мета-теги
- Деплой на Vercel: `https://vitrina-kappa.vercel.app`

Эти компоненты витрины переиспользуются как есть. Меняется источник данных: вместо YAML — Supabase.

## Архитектура

```
[Мастер] → Админка (PWA, /admin/*) → Supabase (Postgres + Storage)
                                         ↓
                                    On-demand revalidation
                                         ↓
[Клиент] → Витрина (/[slug], статика) ← пересборка из Supabase
```

### Ключевые принципы

1. **Витрина остаётся статической.** Когда мастер сохраняет изменения, его страница `/[slug]` пересобирается через `revalidatePath` или `revalidateTag`. Клиенты получают HTML с CDN мгновенно, без живого backend.
2. **Админка — это PWA.** Manifest + service worker. Мастер может «Добавить на главный экран», и админка работает как нативное приложение.
3. **Авторизация — только через Telegram Login Widget.** Никаких email, паролей, SMS.
4. **Один Telegram-аккаунт = одна витрина** (на этом этапе). Если мастер уже создал витрину и пытается создать вторую — система ведёт его в существующую.

## Технический стек (дополнения к существующему)

- **Supabase** (Postgres + Storage + Auth) — основной backend
- **Supabase JS SDK** для работы с базой и хранилищем
- **Telegram Login Widget** — авторизация мастеров
- **`@vercel/og`** — автогенерация OG-картинок (опционально, если будет время)
- **`next-pwa`** или ручной service worker + manifest — для PWA
- **`zod`** — валидация данных форм
- **`react-hook-form`** — формы в админке
- **Vercel Deploy Hook** или `revalidatePath` — обновление витрин при изменениях

## Структура данных в Supabase

### Таблица `merchants`

```sql
create table merchants (
  id uuid primary key default gen_random_uuid(),
  telegram_id bigint unique not null,           -- Telegram user ID мастера
  telegram_username text,                        -- @username (может меняться)
  telegram_first_name text,                      -- для CTA-кнопки
  slug text unique not null,                     -- baba-zina
  name text not null,                            -- Носки от бабы Зины
  tagline text,                                  -- Ручная вязка, г. Пермь
  avatar_url text,                               -- путь в Supabase Storage
  accent_color text default '#854F0B',
  sale_percent int,                              -- если NULL — распродажа выключена
  sale_until timestamptz,                        -- дата окончания распродажи
  sale_text text default 'Распродажа на всё',
  contact_telegram text not null,                -- куда ведёт CTA, по умолчанию = telegram_username
  is_published boolean default false,            -- черновик / опубликована
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create index idx_merchants_slug on merchants(slug);
create index idx_merchants_telegram on merchants(telegram_id);
```

### Таблица `products`

```sql
create table products (
  id uuid primary key default gen_random_uuid(),
  merchant_id uuid references merchants(id) on delete cascade,
  name text not null,
  price int not null,                            -- в рублях, целое число
  description text,                              -- 1-2 предложения
  image_url text,                                -- путь в Supabase Storage
  discount_percent int,                          -- индивидуальная скидка
  is_available boolean default true,             -- в наличии / нет
  sort_order int default 0,                      -- для drag-n-drop порядка (опционально)
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

create index idx_products_merchant on products(merchant_id);
```

### Supabase Storage

Два bucket:
- `avatars` — публичный, для аватаров мастеров
- `products` — публичный, для фото товаров

Структура файлов:
- `avatars/{merchant_id}.jpg`
- `products/{merchant_id}/{product_id}.jpg`

### Row Level Security (RLS)

**Обязательно включить RLS на обеих таблицах.**

Политики:
- `merchants`: чтение публичное (для витрин), запись только своего (где `telegram_id = auth.uid`)
- `products`: чтение публичное, запись только если `merchant_id` принадлежит залогиненному мастеру

## Telegram Login Widget — как это работает

1. На странице `/login` рендерится скрипт Telegram Widget с `data-telegram-login="vitrina_kappa_bot"`
2. Мастер нажимает кнопку «Войти через Telegram»
3. Telegram открывает диалог подтверждения, мастер подтверждает
4. Widget возвращает callback с данными: `id`, `first_name`, `username`, `auth_date`, `hash`
5. Серверная функция (Next.js API route `/api/auth/telegram`) **обязательно проверяет hash** через HMAC-SHA256 от `TELEGRAM_BOT_TOKEN` — иначе можно подделать
6. Если hash валиден — создаёт сессию (через Supabase Auth с custom JWT или через cookies)
7. Редирект в `/admin`

**Документация:** https://core.telegram.org/widgets/login

## Сценарии пользователя

### Сценарий 1: Новый мастер регистрируется

1. Заходит на `/`
2. Видит описание проекта, кнопку «Создать витрину»
3. Тыкает → `/login`
4. Видит Telegram Login Widget, нажимает «Войти через Telegram»
5. Telegram подтверждает, callback приходит на `/api/auth/telegram`
6. Сервер проверяет hash, ищет мастера в базе:
   - Если нет — создаёт запись с `telegram_id`, `telegram_username`, `telegram_first_name`. Редирект на `/admin/setup`
   - Если есть — редирект на `/admin`
7. На `/admin/setup` мастер заполняет название магазина и подзаголовок
8. Система предлагает slug на основе названия (см. ниже)
9. Мастер нажимает «Создать витрину» → запись в `merchants` обновляется, редирект на `/admin/products`
10. Мастер добавляет первый товар
11. Мастер идёт на `/admin/share`, копирует ссылку, рассылает клиентам

### Сценарий 2: Мастер редактирует существующий товар

1. Открыл иконку на телефоне (PWA) → автоматический логин (сессия в cookie)
2. Видит `/admin/products`, список товаров с фото и ценами
3. Тыкает по товару → `/admin/products/[id]`
4. Редактирует поле (например, цену) → нажимает «Сохранить»
5. Запрос в Supabase, обновление поля
6. После успешного сохранения — вызов `/api/revalidate` с `slug` мастера, чтобы пересобрать его витрину
7. Через 5–30 секунд новая цена видна клиентам

### Сценарий 3: Клиент открывает витрину

1. Получает ссылку `vitrina-kappa.vercel.app/baba-zina` в Telegram
2. Тыкает → открывается статическая страница
3. Просматривает каталог, тыкает «Написать мастеру»
4. Улетает в Telegram в чат с мастером по `tg://resolve?domain={contact_telegram}`

## Алгоритм генерации slug (гибрид)

Когда мастер создаёт витрину, при заполнении поля «Название магазина» в фоне:

1. Транслитерируется кириллица в латиницу (`Носки от бабы Зины` → `noski-ot-baby-ziny`)
2. Обрезается до 30 символов, отбрасываются «стоп-слова»: `от`, `для`, `ot`, `for` и т.п. (упрощённо)
3. Получается дефолт: `baba-zina` или `noski-baba-zina`
4. Поле slug показывается мастеру с этим дефолтом, **редактируемое**
5. По мере набора — проверка занятости через Supabase (debounce 500ms)
6. Если занято — показать сообщение «Уже занято, попробуйте `baba-zina-2`» с автопредложением
7. Slug должен соответствовать regex `^[a-z0-9-]{3,40}$` (только латиница, цифры, дефис; 3–40 символов)
8. Запрещённые slug-ы (захардкодить список): `admin`, `api`, `login`, `signup`, `auth`, `static`, `_next`, и все имена системных папок

## Защита от спама / абъюз

1. **Один Telegram-аккаунт = одна витрина.** Уникальный индекс на `telegram_id` в БД. Если мастер пытается создать вторую — редирект в существующую.
2. **Rate limiting** на API-роуты (через Vercel Edge или `@upstash/ratelimit`):
   - `/api/auth/telegram`: 10 запросов в минуту с IP
   - `/api/revalidate`: 60 запросов в минуту с авторизованного юзера
3. **Лимит на размер загружаемых файлов**: фото товара — макс 5 МБ, аватар — макс 2 МБ. Серверная проверка mime-type и реального содержимого (magic bytes).
4. **Лимит товаров на витрину**: 50 шт на этапе 1 (защита от мусорных аккаунтов).
5. **Запрет на удаление аккаунта через UI на этапе 1** (только через прямой запрос в Telegram владельцу проекта — это нормально для теста).

## PWA — что нужно реализовать

1. **manifest.json** в `public/`:
   ```json
   {
     "name": "Витрина — управление",
     "short_name": "Витрина",
     "start_url": "/admin",
     "display": "standalone",
     "background_color": "#FAFAF7",
     "theme_color": "#FAFAF7",
     "icons": [
       { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png" },
       { "src": "/icon-512.png", "sizes": "512x512", "type": "image/png" },
       { "src": "/icon-maskable.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
     ]
   }
   ```
2. **Иконки** 192×192, 512×512, maskable 512×512. Подбери нейтральные «коробочка/магазин» иконки или сгенерируй простым SVG (буква «В» на цветном фоне).
3. **Service Worker** через `next-pwa` ИЛИ ручной минимальный:
   - Кеширование статики (HTML/CSS/JS) — `stale-while-revalidate`
   - Кеширование изображений — `cache-first`
   - **Не кешировать запросы к Supabase API** — данные должны быть свежие
4. **iOS-теги в `<head>`** для корректной работы «На главный экран» в Safari:
   ```html
   <link rel="apple-touch-icon" href="/icon-192.png" />
   <meta name="apple-mobile-web-app-capable" content="yes" />
   <meta name="apple-mobile-web-app-title" content="Витрина" />
   ```
5. **Тестировать установку**: открыть Chrome DevTools → Application → Manifest. Не должно быть ошибок. На реальном телефоне — должна появляться подсказка «Установить» или в меню Chrome — пункт «Добавить на главный экран».

## Дизайн админки — принципы

Это раздел крайне важен — слишком легко скатиться в Tilda-стайл с десятью вкладками. **Админка должна быть для бабушки, которая впервые держит смартфон.**

Принципы:
1. **Один экран — одна задача.** Добавить товар — один экран с полями. Не три вкладки.
2. **Большие пальцы, большие кнопки.** Минимальный touch-target — 48×48 px, лучше 56×56.
3. **Никакого жаргона.** Не «SKU», не «артикул», не «слаг», не «опубликовать», не «вариации». Только: «товар», «цена», «фото», «название магазина», «адрес моей витрины».
4. **Только нужные поля.** Если у товара нет описания — не показывать поле. Если в магазине нет распродажи — не пихать переключатель распродажи на главный экран админки.
5. **Дефолты ко всему.** Цвет акцента, валюта, всё уже настроено. Мастер может изменить, но не должен.
6. **Каждое действие — с подтверждением успеха.** «Сохранено» зелёной полосой внизу 2 секунды. Без неё мастер не понимает, сохранилось ли.
7. **Нижняя навигация — 4 пункта максимум:** Товары, Распродажа, Поделиться, Ещё. Не пять, не семь.

## Обновление витрины при изменениях

После любого сохранения (товар, настройки, распродажа) — серверный код вызывает:

```ts
revalidatePath(`/${merchant.slug}`);
```

Это пересобирает только нужную страницу. Через 5–30 секунд новая версия видна клиентам.

Альтернатива (если revalidatePath не сработает на статическом экспорте) — Vercel Deploy Hook: при сохранении делаем POST на webhook URL, Vercel начинает фоновую пересборку всего проекта (2–4 минуты).

**Предпочтительный путь — `revalidatePath` через `output: 'standalone'` (а не `export`), потому что админка требует серверных функций.** То есть проект больше не чисто статический, он гибридный: страница `/[slug]` рендерится через ISR (Incremental Static Regeneration), а админка — серверные роуты.

**Это ВАЖНОЕ изменение архитектуры.** Удалить `output: 'export'` из `next.config.ts`, перейти на гибридный режим Vercel.

## Безопасность

1. **Telegram hash обязательно проверяется на сервере** через HMAC-SHA256 с `TELEGRAM_BOT_TOKEN`. Без проверки можно подделать любого пользователя.
2. **RLS включён** на обеих таблицах, политики настроены.
3. **Загруженные картинки пересохраняются через Sharp** перед попаданием в Storage: только JPEG/PNG/WebP/HEIC, ресайз до 1080×1080 max, WebP на выходе.
4. **CSP-заголовки** на витрине: только свои скрипты + Yandex Metrika + Supabase.
5. **HTTPS принудительно** через Vercel (он сам обеспечивает).
